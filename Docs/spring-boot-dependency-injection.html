🌱 Spring Boot Dependency Injection (DI) – Complete Notes

1️⃣ What is Dependency Injection?

📌 Definition

Dependency Injection (DI) is a design pattern in which an object receives its dependencies from an external source rather than creating them itself.

In Spring Boot, the Spring IoC Container manages object creation and injects required dependencies automatically.

🎯 Why Dependency Injection?

Without DI:

public class OrderService {

    private PaymentService paymentService = new PaymentService(); // Tight coupling

}

Problems:

  • ❌ Tight coupling
  • ❌ Hard to test
  • ❌ Not flexible
  • ❌ Hard to replace implementations

With DI:

@Service
public class OrderService {

    private final PaymentService paymentService;

    public OrderService(PaymentService paymentService) {
        this.paymentService = paymentService;
    }
}

Advantages:

  • ✅ Loose coupling
  • ✅ Easy testing (mocking)
  • ✅ Flexible design
  • ✅ Clean architecture

2️⃣ Core Concept: IoC (Inversion of Control)

📌 What is IoC?

Inversion of Control means:

The control of object creation is transferred from the programmer to the Spring container.

Instead of:

Your Code → Creates Objects

It becomes:

Spring Container → Creates Objects → Injects into Your Code

  • ApplicationContext
  • @ComponentScan

4️⃣ Types of Dependency Injection in Spring

Type Recommended? Notes
Constructor Injection ✅ YES Best practice
Setter Injection ⚠️ Sometimes Optional dependencies
Field Injection ❌ NO Not recommended

5️⃣ Constructor Injection (Recommended)

📌 Why Constructor Injection?

  • Immutable dependencies
  • Easy unit testing
  • Clear required dependencies
  • Prevents NullPointerException

✅ Example

Step 1: Create Dependency

@Service
public class PaymentService {

    public void processPayment() {
        System.out.println("Payment processed");
    }
}

Step 2: Inject Dependency

@Service
public class OrderService {

    private final PaymentService paymentService;

    public OrderService(PaymentService paymentService) {
        this.paymentService = paymentService;
    }

    public void placeOrder() {
        paymentService.processPayment();
    }
}

🔎 What Happens Internally?

Spring Container:
    1. Creates PaymentService bean
    2. Creates OrderService bean
    3. Injects PaymentService into constructor

6️⃣ Setter Injection

Used when dependency is optional.

@Service
public class NotificationService {

    private EmailService emailService;

    @Autowired
    public void setEmailService(EmailService emailService) {
        this.emailService = emailService;
    }
}

⚠️ Risk: Object can be created without dependency.

7️⃣ Field Injection (Not Recommended)

@Service
public class OrderService {

    @Autowired
    private PaymentService paymentService;
}

Problems:

  • ❌ Hard to test
  • ❌ Reflection-based injection
  • ❌ Cannot make field final
  • ❌ Hidden dependencies

8️⃣ @Autowired – Do We Still Need It?

👉 Since Spring 4.3:

If a class has only one constructor, @Autowired is optional.

@Service
public class OrderService {

    private final PaymentService paymentService;

    public OrderService(PaymentService paymentService) {
        this.paymentService = paymentService;
    }
}

Spring automatically injects the dependency.

9️⃣ Multiple Implementations Problem

Scenario

public interface PaymentService {
    void processPayment();
}

Two implementations:

@Service
public class CreditCardPaymentService implements PaymentService {
    public void processPayment() {}
}

@Service
public class UpiPaymentService implements PaymentService {
    public void processPayment() {}
}

Spring will throw:

NoUniqueBeanDefinitionException

🔧 Solutions

✅ 1. @Primary

@Service
@Primary
public class CreditCardPaymentService implements PaymentService {
}

✅ 2. @Qualifier (Best for Explicit Selection)

@Service
public class OrderService {

    private final PaymentService paymentService;

    public OrderService(@Qualifier("upiPaymentService") PaymentService paymentService) {
        this.paymentService = paymentService;
    }
}

1️⃣2️⃣ Testing with Constructor Injection

@ExtendWith(MockitoExtension.class)
class OrderServiceTest {

    @Mock
    private PaymentService paymentService;

    @InjectMocks
    private OrderService orderService;

}

No Spring context required.

1️⃣3️⃣ Common Interview Questions

Q1: Why is constructor injection preferred?

  • Immutability
  • Clear dependencies
  • Better testing
  • Avoid circular dependencies

Q2: What happens if two beans of same type exist?

  • NoUniqueBeanDefinitionException
  • Use @Primary or @Qualifier

Q3: What is circular dependency?

ServiceA → ServiceB
ServiceB → ServiceA

Constructor injection will fail.

1️⃣4️⃣ Best Practices

  • ✅ Always prefer constructor injection
  • ✅ Keep dependencies minimal
  • ✅ Avoid field injection
  • ✅ Use @Qualifier when needed
  • ✅ Use interfaces, not concrete classes
  • ✅ Follow SOLID principles
